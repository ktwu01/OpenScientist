## When
Designing a skill library / tool-orchestration layer for an LLM agent platform. Tempted to support nested skills (skill A calls skill B calls skill C) for code reuse and abstraction elegance.

## Decision
**Build a flat, validated skill library of depth 1 — skills may call atomic tools, but skills must not call other skills.** Each skill is independently testable. SkillCraft measured this: flat skills keep GPT-5.2 at 90% success; adding skill-on-skill composition drops it to 79% due to error propagation through layers with no error-boundary contracts. The error-propagation mode is dominated by null/type/edge cases that the top-layer skill never anticipated because it didn't author the bottom-layer contract.

Rejected — "deep hierarchical skills for better abstraction": elegance at the cost of robustness; the null-temperament dog-breed cascade is the canonical failure.

Rejected — "let agents build whatever composition they want": Minimax-authored skills hurt executors by 48% tokens; letting weak-creator skills enter the library degrades everything downstream. Gate every skill on a quality check.

## Local Verifiers
- Is every skill in your library independently runnable without another skill? If not, flatten.
- Does every skill have a verifier (three-stage check: syntax → runtime → output shape, à la SkillCraft's Coding Verifier)? If not, you can't tell good from bad skills at ingest time.
- Was the skill written by the strongest model you can afford? Creator quality > executor quality (SkillCraft cross-model matrix); don't let cheap models author shared library entries.

## Failure Handling
If a use case truly requires composition, wrap the composition *in code the agent writes per-task* rather than as a persisted library skill. The composition's risk stays scoped to the one task rather than polluting the shared library.

## Anti-exemplars
- A "build macro-skills that call mini-skills" architecture — reads as clean; fails in production on edge cases.
- A skill library where anyone's model can contribute entries — the weakest creator bounds the quality of the whole library.
- A flat library where every skill is wrapped in try/except and returns structured errors — correct pattern.
